📐 Архитектурные стили
Могу обосновать выбор монолита или микросервисов для конкретного проекта
Вопрос: При каких условиях монолит предпочтительнее микросервисов?
→ Малая команда (до 10 человек), простая/единая бизнес-логика, требуется быстрый старт, ограниченный бюджет на инфраструктуру.
📋 Работа с требованиями
Умею собирать и анализировать нефункциональные требования, переводить бизнес-цели в архитектурные драйверы
Вопрос: Бизнес говорит: "Система должна быть быстрой". Какие уточняющие вопросы вы зададите, чтобы превратить это в архитектурное требование?
→ Быстрая для кого? (средний пользователь / VIP-клиенты)
→ Какие операции? (загрузка страницы / обработка платежа)
→ Как измерить? (95-й перцентиль < 500мс)
→ При какой нагрузке? (1000 RPS / 100 одновременных пользователей)
🛡️ Устойчивые решения
Знаю тактики отказоустойчивости, резервирования и мониторинга для надёжных систем
Вопрос: Сервис A вызывает сервис B по HTTP. Сервис B начал отвечать с задержкой 30 секунд. Какие 2 тактики защитят сервис A от каскадного отказа?
→ Timeout (ограничение времени ожидания, например, 3 сек)
→ Circuit Breaker (прекращение запросов после N неудачных попыток)
→ Bulkhead (изоляция пула потоков/соединений для разных зависимостей)
⚠️ Важно: Retry при систематических задержках может усугубить проблему! Retry уместен только при спорадических сбоях (сетевые глитчи), а не при перегрузке сервиса.
🔗 Интеграция микросервисов
Понимаю разницу между синхронными (REST/gRPC) и асинхронными (брокеры, очереди) взаимодействиями
Вопрос: Когда стоит выбрать асинхронное взаимодействие через брокер сообщений вместо синхронного REST/gRPC?
→ Когда не нужен немедленный ответ (отложенная обработка)
→ Когда нужна гарантия доставки + переотправка при сбоях
→ Когда хотим развязать сервисы (отправитель не знает получателей)
→ Примеры: отправка email, обработка заказов, аналитика событий
🧩 DDD и ограниченные контексты
Применяю тактическое и стратегическое моделирование для сложной предметной области
Вопрос: У вас есть сущность "Пользователь". В контексте "Аутентификация" и в контексте "Доставка заказов" — это одна модель или разные? Почему?
→ Разные модели (разные bounded contexts)
→ В "Аутентификации": важны email, пароль, роли
→ В "Доставке": важны адрес, телефон, история заказов
→ Попытка сделать единую модель → bloated entity со всеми полями
